# Why Next.js? (프론트엔드 렌더링의 진화)




Next.js는 아래를 만족하기 쉬운 React 애플리케이션을 쉽게 만들기 위해 등장했다.

- **검색 엔진에 잘 노출되면서도(SSR)**
- **높은 초기 로딩 속도를 가지고**
- **사용자 경험이 부드러운(CSR)**



## 1. 과거: MPA (Multi Page Application)

초기 웹 개발 방식입니다. (PHP, JSP, 초기 ASP.NET 등)

*   **동작 방식**: 사용자가 페이지를 이동할 때마다 서버에서 **새로운 HTML을 완성해서** 보내줍니다.

*   **장점**:
    *   서버에서 완성된 HTML을 주므로 **SEO(검색 엔진 최적화)**에 유리합니다.
    *   첫 화면 로딩(FCP)이 빠릅니다.

*   **단점**:
    *   페이지 이동 시마다 화면이 깜빡입니다 (새로고침).
    *   사용자 경험(UX)이 모바일 앱에 비해 좋지 않습니다.



## 2. 혁명: SPA (Single Page Application)와 CSR

React, Vue, Angular의 등장으로 웹 개발의 패러다임이 바뀌었습니다.

*   **동작 방식 (CSR - Client Side Rendering)**:
    1.  서버는 텅 빈 HTML(`div id="root"`)과 거대한 JavaScript 파일만 보냅니다.
    2.  브라우저가 JS를 다운로드하고 실행한 뒤, 클라이언트(브라우저)에서 화면을 그립니다.

*   **장점**:
    *   페이지 이동 시 깜빡임이 없고 부드럽습니다 (앱 같은 UX).
    *   필요한 데이터만 서버와 통신하므로 서버 부하가 줄어듭니다.

*   **단점 (Next.js의 등장 배경)**:
    1.  **SEO 문제**: 검색 엔진 크롤러가 사이트에 방문했을 때 텅 빈 HTML만 보게 되어 내용을 수집하지 못하는 경우가 많았습니다.
    2.  **초기 로딩 속도(TTV) 저하**: JS 번들 파일이 다 다운로드되고 실행될 때까지 사용자는 흰 화면만 봐야 합니다.



## 3. 해결책: Next.js (The Hybrid)

Next.js는 React의 문법과 SPA의 장점을 유지하면서, 서버 사이드 렌더링(SSR)을 도입해 위 문제들을 해결했습니다.

*   **동작 방식 (Hydration)**:
    1.  **Pre-rendering**: 서버에서 미리 HTML을 만들어 브라우저에 보냅니다. (사용자는 즉시 내용을 볼 수 있음 - SEO/초기로딩 해결)
    2.  **Hydration**: 이후 JS가 로드되면 HTML에 '물(기능)을 채우듯' 연결되어 React처럼 작동합니다. (SPA의 장점 유지)


### Next.js가 제공하는 렌더링 전략

Next.js는 상황에 따라 렌더링 방식을 선택할 수 있게 해줍니다.

1.  **SSR (Server-Side Rendering)**: 요청할 때마다 서버에서 HTML 생성 (최신 데이터 필요 시)
2.  **SSG (Static Site Generation)**: 빌드 타임에 HTML 생성 (블로그, 문서 등 변하지 않는 내용)
3.  **ISR (Incremental Static Regeneration)**: 주기적으로 정적 페이지를 업데이트
