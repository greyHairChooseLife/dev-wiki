# Pintos 회고


**TOC**

- OS에 대한 이해
    - booting
        - 물리메모리
        - 가상주소공간(Page Table)
    - 쓰레드 생성
    - 동시성
        - mutex, semaphore, conditional variable
    - schedule
        - timer interrupt
        - context switch
    - interrupt setup
    - system call
    - paging

    - (곁)malloc
    - 페이지 테이블


- 디버깅 경험
    
    - register, memory 레벨까지 내려가는 디버깅
    - gdb with pwndbg 재미있다.
    - objdump로 ELF 파일을 확인하니 명확해지는 것들이 있다.


- 시스템 프로그래밍
    
    - 변수를 선언한다 => 메모리를 한 자리 차지한다.
    - 변수의 초기화를 한다/안한다 => 가상메모리의 어느 영역을 차지한다.
    - 컨텍스트 스위치같은 결정적인 제어는 assembly를 이용해 레지스터를 직접 다뤄야한다.


- 컴파일, 실행파일

    - 컴파일 하는 시점에 이미 가상주소공간을 어떻게 활용할지 정해진다.
    - 그러니까 어떤 실행환경(아키텍쳐)에서 실행될지 컴파일(링킹) 전에 알아야 한다는건 지극히 당연한
      말이다.



- 개발환경

    - tmux 및 tmuxp를 활용한 개발환경 관리는 매우 효과적이다.
        - tmuxp 명세는 yaml파일이며, 그 자체로 문서의 기능을 한다.
        - 다만, 개인적으로만 사용하는 alias 사용을 지양하자. 시스템이 바뀌었을 때 재현이 어렵다.
        - 주석을 적극적으로 달자.



- 협업

    - history 관리
        - PR의 volume을 적절히 계획하는 것이 무엇보다 중요하다.
        - PR을 위한 브랜치는 별도로 만들 수도 있다.
        - 개발 과정, 사고의 흐름을 그대로 표현하는 history도 좋지만, 팀원이 모든 것을 알아야하는 것은 아니다.

    - 다른 사람의 코드는 더 잘 보인다.
        - 물길이든 사람 다니는 길이든 일단 길이 나면 길로 다니게 된다. 우리가 생각하는 방식도
          마찬가지다.
        - 동료와 힘을 합치면 새로운 시각에서 보기 쉽다.

    - 설명을 해보거나 설명을 듣고 이해해보는 과정을 통해 사고력, 집중력을 훨씬 더 발휘하게 되고,
      이해가 부족한 부분을 쉽게 찾아낼 수 있다.
