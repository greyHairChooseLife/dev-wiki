# `<a>` tag vs `<Link>` tag




사용자님이 관찰하신 두 가지 현상은 Next.js의 `<Link>` 컴포넌트가 가진 가장 강력한 기능인 **클라이언트 사이드 라우팅(Client-side Routing)**과 **프리페칭(Prefetching)** 덕분입니다.

이것이 바로 Next.js가 "첫 로딩은 SSR이지만, 이후에는 SPA처럼 동작한다"고 말할 수 있는 기술적 근거입니다.

### 1. 왜 네트워크 요청을 다시 하지 않는가? (Client-side Navigation)

`<a>` 태그와 `<Link>` 컴포넌트의 결정적인 차이는 **브라우저의 동작 방식**에 있습니다.

*   **`<a>` 태그 (Hard Navigation):**
    *   브라우저는 현재 페이지를 완전히 **버리고(Unload)**, 서버에 새로운 URL에 대한 HTML 문서를 처음부터 다시 요청합니다.
    *   CSS, JS 파일도 다시 로드하고 실행해야 하므로 화면이 깜빡이고 느립니다.

*   **`<Link>` 컴포넌트 (Soft Navigation):**
    *   브라우저는 페이지를 새로고침하지 않습니다.
    *   자바스크립트(Next.js 라우터)가 URL만 변경(`history.pushState`)합니다.
    *   **JSON 데이터만 교체:** 서버로부터 전체 HTML을 받는 것이 아니라, 다음 페이지를 그리는 데 필요한 **최소한의 데이터(RSC Payload 또는 JSON)**만 가져와서 현재 화면의 내용을 갈아 끼웁니다.
    *   이미 방문했던 페이지라면, Next.js는 메모리 내의 **Router Cache(클라이언트 캐시)**에 저장된 데이터를 재사용하므로 네트워크 요청조차 발생하지 않을 수 있습니다.

### 2. 왜 미리 받아오는가? (Prefetching)

사용자님이 느끼신 "마치 미리 받아온 것 같은" 느낌은 실제로 **미리 받아왔기 때문**입니다.

Next.js의 `<Link>` 컴포넌트는 뷰포트(화면)에 나타나는 순간 자동으로 동작합니다.

1.  **감지:** 사용자의 화면(Viewport)에 `<Link href="/about">`이 보입니다.
2.  **백그라운드 요청:** Next.js는 사용자가 아직 클릭하지 않았음에도, "사용자가 저기로 갈 확률이 높다"고 판단하고 백그라운드에서 조용히 `/about` 페이지에 필요한 데이터(JS, JSON)를 미리 다운로드합니다.
3.  **즉시 이동:** 사용자가 실제로 링크를 클릭하는 순간, 이미 데이터가 다운로드되어 있으므로 **지연 시간 없이(0ms)** 즉시 페이지가 전환됩니다.

### 요약: `<a>` vs `<Link>` 비교

| 기능 | `<a>` 태그 (As Is) | `<Link>` 컴포넌트 (To Be) |
| :--- | :--- | :--- |
| **동작 방식** | 전체 페이지 새로고침 (MPA 방식) | 자바스크립트로 내용만 교체 (SPA 방식) |
| **네트워크** | HTML, CSS, JS 전체 다시 요청 | 필요한 데이터(JSON/JS)만 요청 |
| **속도** | 화면 깜빡임 발생, 상대적으로 느림 | 깜빡임 없음, 즉시 전환 |
| **스마트 기능** | 없음 | **Prefetching** (화면에 보이면 미리 로드) |
| **캐싱** | 브라우저 기본 캐시 의존 | **Router Cache** (방문한 페이지 메모리 저장) |
