- 우리끼리 회의를 했다. 우리는 어디로 어떻게 나아가고 있는가?
    - 서로 하고있는 일을 합치자

    - PR/리뷰라는 과정은 무엇인가? 어떻게 할 것인가?

    - 나만무를 시작할 때 AI agent로 어느정도 코드를 만들고, 그것일 이해하고 활용하는 방식을 제안했었는데 그땐 그냥 떠오른 아이디어 정도였다.
        - 팀원들도 별 반응이 없어서 그냥 지나가는 아이디어 정도였다.
        - 그런데 이제보니 결과적으로 그렇게 해보게 되었다.

    - 각자는 무엇을 했는지도 이야기 나눴다. 나는 뭘 했을까?
        - 개발 환경 세팅
            - HTTPS 프로토콜을 지원하면서도 localhost에서 개발 서버를 띄울 수 있도록 어떻게든 해보려고 하다가 실패했다.
                - cloudflare tunnel로 거의 성공했는데, 이 서비스는 우리에게 필요한 UDP 프로토콜을 지원하지 않았다.
            - EC2 인스턴스를 만들고 보안 규칙을 적용해서 우리에게 필요한 포트와 프로토콜을 적절히 노출시켰다.
            - 이것에 안전하고 쉽게 접근할 수 있도록 쉘 스크립트를 작성하여 팀원들에게 배포했다.
        - 개발 서버 세팅
            - 우리팀은 데모를 만들어본 소스코드를 FE/BE로 레포지토리 단위로 분리했다.
            - 원본 소스코드는 mono-repo인데, 팀원분들은 docker-compose 등이 낯설어서 순탄한 개발환경 세팅에 어려움을 겪고 계셨다.
                - 필요 없는 파일들 및 환경변수를 종합했다.
                - BE에서 docker-compose 하나만 실행하면 모든 것이 완료되도록 하였다.
                - 빨리 배포하려다보니 PR 과정이라던가 그 각종 규칙이 많이 번거롭게 느껴졌다.
        - terraform 테스트
            - 익화님의 가볍게 추천해 주셨다. 처음들어보는거였다.
            - infra resources를 선언적으로 관리하는 도구다.
                - aws뿐 아니라 여러 cloud infra providers를 지원한다.
                - 우테코였나? 그 영상 중에 끝내주게 설명을 잘 해주신게 있어서 쉽게 익힐 수 있었다.
            - ec2 인스턴스와 elastic IP와 연계해서 켜고 끄는 작업을 조금 테스트해봤다.
                - 목표로 하는 셋업이 있으면 금방 할수있을듯 하다.
            - 테라폼 이거 물건이다.
            - ansible이라는 것도 있는데, 해보진 않았지만 뭘 미리 설치한다던가 하여튼 initial 작업이
              많아지면 아주 유용하겠다. 거의 필수일 것 같다.
        - Continuous Integration: AI 리뷰 / Pull_Request
            - CI의 한 단계로써 AI 리뷰를 만들어봤다.
            - 정글에서 지원해준 크레딧으로 aws bedrock을 사용했다.
            - github action은 브랜치별로 반영되는지 어떤지 헷갈리는 부분들이 좀 있었는데, 이 과정에서 많이 명쾌해졌다.
        

        - 그리고 브랜치관리 전략을 재고해보고, 문서화하여 공유하기로 했다.
